# Shadowfax Unified API for Forward Integrations

You can integrate with Shadowfax's Last Mile Customer Delivery Platform by using the Unified API Collection. Unified APIs provide a singular interface to enable Marketplace Seller and Warehouse models of operations. These APIs support end to end order lifecycle starting from Pincode serviceability to order status tracking until delivery.
## Glossary

###DCs and Hubs
*DC*: Large facilities typically located in the outskirts of a city where all shipments(intercity and intracity both) are processed.<br>


*Hub*: Small facilities located across the city from where last-mile delivery and first mile pick up happens. 


####Following terms are defined concerning a shipment's location
* Origin Hub:<br>
This is the first Shadowfax facility where a packet reaches directly from the seller or customer
* Origin DC:<br>
This is the first DC where a packet is sent from the client Warehouse or origin hub to process further.
* Destination Hub:<br>
This is the last shadowfax facility where a packet reaches before delivering to customer or seller
* Destination DC:<br>
This is the last DC where a packet is sent before sending to the destination hub or client Warehouse
* Via DC:<br>
All DC's, other than origin and destination where a shipment has to go through. Manifest bags are not opened at these DCs and will be directly passed on to the next location.

####Type of Operations
* Forward Operations:<br>
Orders that are originating at client warehouse or seller side and are to be delivered to end customer.<br>
<br>
* Reverse Operations: <br>
Return orders that are originating at the customer side or forward orders that could not be delivered to the customer and are to be returned to the client warehouse or seller. <br>

####Miscellaneous
<br>
* AWBs: <br>
Unique Identification Number for each item/packet <br>
* LOR: <br>
Letter of Receipt, a document for acknowledging the delivery or receipt of shipments/ packets <br>
* POD(Proof of Delivery): <br>
A receipt of pickup and delivery of items

##Responses
Shadowfax API uses HTTP status codes to indicate the status of your requests. This includes successful and unsuccessful responses.

* All responses are in [JSON]('http://www.json.org') by default.
* All response Timestamps follow [ISO-8061]('https://www.w3.org/TR/NOTE-datetime') format.
    > Times are expressed in UTC (Coordinated Universal Time), with a special UTC designator ("Z").
    
#### Status Codes

* 200 - OK: Everything went as planned.
* 400 - Bad Request: You did something wrong. Often a missing argument or parameter.
* 401 - Unauthorized: Authentication was incorrect.
* 404 - Not Found
* 500 - Internal Server Error: We had a problem processing the request.
* 503 - Service Unavailable: Try again later.

##Push Callback API
We have built a generic PUSH API callback for both Marketplace(seller pickup) and warehouse models. The client can enable it to get real-time notification of each push notification status change.

### Callback Header details

The Standard callback header has the following keys: `Authentication Key(Non mandatory).`


+ Headers

      Authorization: Token c6b780ae9250f3
      Content-Type: application/json

Here we have the capability to add/modify the header keys as shown in the example below:

+ Headers

    Content-Type: application/json
    Authorization: d5eecf30ae6eb8d158dc54c77178ab9
    account_name:  SF_ACC_STAGING


### Request Body details
The callback contains the below parameters:

* `awb_number` - Generated by Shadowfax while placing an order.
* `order_id` - client_order_id shared while placing an order.
* `event_timestamp` - event_timestamp of the current status.
* `current_location` - The current location of the order during the event.
* `comments` - Contains the remarks associated with the status.
* `status` - Current status of the shipment
* `event` - Current status id of the shipment
* `otp_verifed` - The value indicates whether the status was marked after OTP verification. The possible values are `"Y"/"N"/"NA"`. By Default `"NA"`.
* `rider_name` - The name of the rider delivering the order.The rider details are shared when the current status is Out For Delivery (ofd).
* `rider_contact` - The contact details of the rider delivering the order.The rider details are shared when the current status is Out For Delivery (ofd).
* `type` - The order type is shared over this field. eg: FWD/REV 
* `recepient_info` - This field contains proof of delivery, and it will be sent only when the current status has been delivered or RTS_D has been confirmed.


This is the structure of the payload data(JSON) that will be pushed to your end.

+ Response 200 (application/json)

        {
            "awb_number": "SF610198449AAA",
            "order_id": "FL20230517311447364",
            "event_timestamp": "2023-05-18 16:22:13",
            "current_location": "CHN_Ambattur_EXP",
            "comments": "Item Delivered at CHN_Ambattur_EXP",
            "event": "delivered",
            "status": "Delivered",
            "otp_verified": "Y",
            "rider_name": null,
            "rider_contact": null,

            "recipient_info": {
                "recipient": "Customer",
                "recipient_name": "Chittaranjan",

When we want to send a push message to your server, we need to make an API call to a push service. API call would require the following callback details, which can be configured via the Client Portal on the Webhook tab.
#### Note : Customized Request Body for Push Callback

We also have the capability to send the below optional fields on clients request as an addition to the above request body :
    
* `updated_date` - Here the value would be the event updated date and the format would be `YYYY-MM-DD` eg: `2023-05-18`
* `updated_time` - Here the value would be the event updated time. The format would be `HHMM`. eg: `1622`    
* `created_date` - Here the value would be the order created date. The format would be `YYYY-MM-DD` eg: `2023-05-18`. 
* `created_time` - Here the value would be the order created time. The format would be `HHMM`.  eg: `1622`
* `total_amount` - This field would be the total amount (inclusive of GST) passed during order placement. 
* `destinationPincode` - This field would have the delivery pincode.
* `destinationCity` -  This field would have the drop city name. default: `null`
* `originCity` - This field would have the drop city name.
* `shipmenttype` - This field indicated the order leg, that is for reverse leg the value would be `R` and for forward leg the value would be `F`. This field is not the same as the `type` field.
* `payMode` - Here the value indicates the payment mode. The value would be `C` for cod mode and `P` for prepaid mode.
* `codAmount` - This field would have the cod amount.
* `estimatedDeliveryDate` - This field would have the promised date. The format would be `YYYY-MM-DD HH:MM:SS`. 
* `attempt_number` - This field would have the attempt count. default: 0

## Unified Pincode Serviceability [/v1/clients/serviceability/?service={service}&page={page}&count={count}&pincodes={pincodes}]
### List All Serviceable Pincodes  [GET]

By using this endpoint, you can get a list of all Shadowfax serviceable pincodes, as well as the services provided by each of them.

If pagination is not sent as a parameter then by default 25 pincodes will be retrieved.

While triggering the API enter the service for which the list of pincode needs to be fetched.
* The service parameter can be:
    * `seller_pickup` - List all the pincodes where seller pickup is available.
    * `customer_delivery` - List all the pincodes where customer delivery is available.
    * `customer_pickup` - List all the pincodes where customer pickup is available.
    * `seller_delivery` - List all the pincodes where seller delivery is available.
    * `warehouse_pickup` - List all the pincodes where warehouse pickup is available.
    * `warehouse_return` - List all the pincodes where warehouse return or rto is available.

We have an automated process of sharing new and unused AWBs once the existing ones are about to exhaust. 
It takes the count of AWB Number as the parameter. The endpoint automatically generates new AWBs and returns them if the unused AWBs are less than the required count.

# group Update Order Details

## Update Order Data  [/v3/clients/order_update/]

### Request to Update Order Data [POST]
The endpoint allows you to edit delivery_details, pickup_details,return_details and order status for an order.

The API functionalities are as follows:

* This API can be used to update the order status as `rts` / `rto` / `reopen_ndr` only if the current status of the order is either `on_hold` or `cancelled_by_customer` 
* The API can be used to update customer details along with the Pincode, but the Pincode can only be updated twice per order and only for intra-city orders.
* The mandatory/required attributes should be passed as `null` if they do not need to be updated.
* The Pincode cannot be updated when the current status of the order is `delivered` .
* This API can be used to update the payment mode of the order to prepaid if the `cod_amount` is passed as 0, but the updation can only be done after the pickup phase i.e post the item is picked.
* The shipment weight can be updated only once, and only while the shipment is in new status.
* The delivery_details can be updated before OFD, the return_details can be edited until the order is cancelled_by_customer and pickup_details can be edited until the order is assigned_for_seller_pickup.

<br><br>The `Authorization` header that should be used while sending request will look like:
<br>
`Token <auth token here>` 
<br>The `Authorization` value itself contains the Word “Token”.

#group Order Cancellation
## Order Cancellation [/v3/clients/orders/cancel/]

### Cancel a Request [POST]

This endpoint cancels marketplace or warehouse orders using the AWB Number or unique client order id, along with the cancellation reason. 

For successful cancellation, we give a Response message as “Request has been marked as cancelled” & 200 as response code.

For below mentioned statues, the API queues the cancellation request, returning 'Request is queued for cancellation' & 304 as response code.
* Out for pickup (ofp)
* received at reverse hub (recd_at_rev_hub)
* Out for delivery (ofd)
* Bag In Transit (bag_in_transit)
* Bag Received at Via (bag_received_at_via)

The cancellation is executed when the shipments are received in the next SFX facility. In case the shipment is “Delivered”, then the canceling request is not executed.

Cancellation after delivery results in failure, returning 'Order cannot be cancelled. Invalid state.' & 400 as response code.

                "cancel_remarks": "Request cancelled by customer"

            + cancel_remarks(required , string) - Valid options are discussed in the description

This endpoint is used  take escalations from clients in real time. Based on the issue category, a ticket would be raised in the system. This API is to take the escalation input from the clients. The action being taken on those tickets .

####Issue Category options 

|Issue Category                        |Issue Category Code  |Validations                                                                                                     |  Order Type           |
|--------------------------------------|---------------------|----------------------------------------------------------------------------------------------------------------|---------------------|
| `Delayed Delivery`                   | 1                   |To be passed post an order is successfully picked in system and not delivered post to Estimated Delivery Date   |    Forward          |
| `Expedite Pickup - Customer`         | 2                   |To be passed only if the order is not yet picked from the Customer                                              |    Reverse          |
| `Expedite Pickup - Seller`           | 3                   |To be passed only until the seller pickup is not completed                                                      |    Forward          |
| `Status Mismatch`                    | 4                   |To be passed when there is a mismatch in expected and received order status                                     |    Forward/Reverse  |
| `Delivery Dispute`                   | 5                   |To be passed only post the order is marked Delivered in the system                                              |    Forward          |

        + Required Parameters(required, object)

            + issue_category(required , number) - Issue category will be one among choices given in Issue Category options .
